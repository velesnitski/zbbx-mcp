# ADR 066: Clear four Dependabot CVEs (cryptography, starlette, pydantic-settings)

**Status:** Accepted
**Date:** 2026-06-23

## Problem

Dependabot raised four alerts at once, all against **transitive** pins in
`uv.lock` (every one pulled in via `mcp`, none declared in
`pyproject.toml`):

| Package | Advisory | Sev | From → To | Issue |
|---------|----------|-----|-----------|-------|
| cryptography | [GHSA-537c-gmf6-5ccf](https://github.com/advisories/GHSA-537c-gmf6-5ccf) | High | 46.0.7 → 49.0.0 | Vulnerable OpenSSL statically bundled in the project's PyPI **wheels** (fixed 48.0.1) |
| starlette | CVE-2026-54283 | High | 1.2.1 → 1.3.1 | Oversized `application/x-www-form-urlencoded` body → DoS (fixed 1.3.1) |
| starlette | CVE-2026-54282 | Low | 1.2.1 → 1.3.1 | Unvalidated request path concatenated into the authority poisons `request.url.hostname` (fixed 1.3.0) |
| pydantic-settings | [GHSA-4xgf-cpjx-pc3j](https://github.com/advisories/GHSA-4xgf-cpjx-pc3j) | Moderate | 2.13.1 → 2.14.2 | `NestedSecretsSettingsSource` follows symlinks out of `secrets_dir` (out-of-tree file read, bypasses `secrets_dir_max_size`) when `secrets_nested_subdir=True` (fixed 2.14.2) |

Reachability mirrors the earlier dependency CVEs (ADR 054/064/065): the two
**starlette** issues are HTTP-request-parsing bugs reachable only under the
**SSE / streamable-http** transports — the default **stdio** deployment
never parses HTTP requests. **pydantic-settings** is only affected when an
app opts into `secrets_dir` with nested subdirs; zbbx-mcp reads
configuration straight from environment variables (`config.py`), never via
that loader. **cryptography**'s bundled-OpenSSL issue only matters when the
vulnerable OpenSSL code path is exercised. None are reachable in this
server's common setup — but four flagged CVEs (one High) in a public repo's
lockfile are cleared regardless.

(starlette was last bumped in ADR 054, 1.0.0 → 1.2.1; this takes it to the
current fix release.)

## Decision

Re-resolve the three packages in one shot, lockfile-only:

```
uv lock --upgrade-package cryptography \
        --upgrade-package starlette \
        --upgrade-package pydantic-settings
```

Resolved to cryptography 49.0.0, starlette 1.3.1, pydantic-settings 2.14.2
— each at or above its fix floor, all the latest releases `mcp`'s
constraints admit. Batched into one commit/ADR/version bump because they
landed as a single Dependabot notification and are the same class of
change; four separate releases would be ADR/version churn for no gain.

## Test approach

No code changed. cryptography is a three-major jump (46 → 49), so the full
suite is the compatibility evidence: 608 tests (incl. `test_server.py`'s
JSON-RPC subprocess boot) pass after `uv sync` on the new set; the server
still builds and registers all 162 tools.

## Consequences

- Tool count unchanged (162). Test count unchanged (608).
- `uv.lock` only (plus the version bump); no source, no API change.
- All four alerts clear once `uv.lock` lands on the default branch.

## Not included

- **Direct version floors in `pyproject.toml`.** Not added — the bumps are
  transitive and `uv` already prefers the newer releases on re-resolve.
- **Pre-empting the rest of the HTTP/auth subtree.** These keep surfacing
  from the optional transport stack `mcp` pulls in. A periodic
  `uv lock --upgrade` (whole-tree, ahead of Dependabot) is the structural
  answer; deferred to an explicit maintenance pass.
